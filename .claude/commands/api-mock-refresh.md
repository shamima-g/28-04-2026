# /api-mock-refresh

Run this command when your backend developer has updated the API spec and you need the mock handlers to reflect the changes.

## What this does

- Detects what changed between the current spec and the snapshot from the last generation
- Updates the canonical spec if the source spec has changed
- Regenerates only the affected handlers — hand-curated data for unchanged endpoints is preserved
- Updates `mock-spec-snapshot.yaml` and `mock-data-context.md`

## Prerequisites

- Mock infrastructure must already be set up (i.e. `web/src/mocks/` exists and `mock-spec-snapshot.yaml` exists in `generated-docs/context/`)
- If mock setup hasn't been done yet, this is not the right command — mock setup is part of the DESIGN phase

---

## Steps

### Step 1 — Read Prerequisites

Read all five files in parallel:
- `generated-docs/specs/api-spec.yaml` — canonical spec
- `generated-docs/context/mock-spec-snapshot.yaml` — snapshot from last generation
- `generated-docs/context/mock-data-context.md` — data conventions
- `generated-docs/context/intake-manifest.json` — intake manifest (needed for Step 2)
- `web/src/mocks/handlers.ts` — existing handlers

If any read fails (file not found), stop and tell the user: "Mock infrastructure doesn't appear to be set up yet. This command is for updating an existing mock layer. Mock setup runs automatically during the DESIGN phase when you choose 'Backend API in development' as your data source."

### Step 2 — Check for Source Spec Updates

Check if the intake manifest (read in Step 1) has `artifacts.apiSpec.userProvided` set. This field is `null` or an **array** of `{ "path", "note" }` objects (P9 — supports multiple specs). If the array is non-empty, use the first entry's `path` as the primary source spec. If multiple entries exist, read all of them — the canonical `api-spec.yaml` was merged from all of them during DESIGN, so use the first as the primary comparison source. If `userProvided` is `null`, fall back to scanning `documentation/` for any `.yaml` or `.json` file with OpenAPI/Swagger content.

If no source spec is found (common when the spec was fully generated by design-api-agent), skip to Step 3.

If a source spec is found, read it and compare its contents against the canonical `generated-docs/specs/api-spec.yaml` (already in memory from Step 1).

If they differ, ask the user using `AskUserQuestion`:
- "I found an updated spec at `documentation/<spec-file>`. Should I update the canonical spec from it before refreshing the mock handlers?"
- Options: "Yes, update from source" / "No, use the existing canonical spec as-is"

If "Yes": reconcile `x-source` tags and adopt the user's schema into the canonical spec:

1. For each endpoint in the user's source spec: set `x-source: user-provided` in the canonical spec and adopt the user's schema (request params, request body, response schema). This applies to both new transitions (`agent-inferred` → `user-provided`) and existing `user-provided` endpoints whose schema has changed.
2. For endpoints in the canonical spec that are NOT in the user's source spec: keep them with `x-source: agent-inferred` (still needed for mock coverage).
3. For endpoints that transition from `agent-inferred` → `user-provided`: if the schema changed, flag this in the changeset summary so the user knows the mock contract shifted.
4. For endpoints already `user-provided` where the schema changed: flag as "schema updated" in the changeset.

Write the reconciled result to `generated-docs/specs/api-spec.yaml`.

### Step 3 — Diff Snapshot vs Current Canonical Spec

Read both files and compare their `paths` sections to produce a changeset.

**Endpoint matching rules:**

- **Match key:** HTTP method + normalized path template
- **Parameter name normalization:** `GET /v1/users/{id}` and `GET /v1/users/{userId}` are the **same endpoint** (parameter names differ but path shape matches). Normalize by replacing all `{paramName}` segments with `{_}` before comparing.
- **Trailing slashes:** Ignored — `/users/` matches `/users`
- **Query parameters:** NOT part of the match key (compared as schema changes within a matched endpoint)

**Diff categories:**

| Category | Definition | Handler action |
|----------|-----------|---------------|
| **Added** | In canonical spec but not in snapshot | Generate new handler |
| **Removed** | In snapshot but not in canonical spec | Remove handler |
| **Schema changed** | Same method+path, but request params, request body, or response schema differs | Regenerate handler (preserve custom data where possible) |
| **Metadata only** | Same method+path+schema, but `x-source`, `x-requirements`, or `description` changed | No handler change needed |
| **Unchanged** | Identical in both | Skip |

Produce the changeset:

```
CHANGESET
---
added:
  - [METHOD] [path]
changed:
  - [METHOD] [path] — [what changed: request params, response schema, etc.]
removed:
  - [METHOD] [path]
metadata_only:
  - [METHOD] [path] — [what metadata changed, e.g. "x-source: agent-inferred → user-provided"]
unchanged:
  - [count] endpoints unchanged
```

If there are no differences (or only metadata-only changes), tell the user: "No handler-affecting changes detected between the current spec and the last snapshot — handlers are already up to date." For metadata-only changes, still update the snapshot (Step 4) so drift detection stays clean. Stop if no handler changes are needed.

### Step 4 — Run mock-setup-agent and type-generator-agent

**Step 4a — Run mock-setup-agent (Call A, partial mode):**

Invoke the `mock-setup-agent` with this prompt:

> "This is Call A — Partial Handler Refresh. This is a /api-mock-refresh run, not an initial generation.
>
> Changeset from spec diff:
> [paste changeset from Step 3]
>
> Apply your partial-refresh rules (Step 2 in your instructions) to update only the affected handlers.
> Update mock-spec-snapshot.yaml with the current canonical spec. Add a `# Generated: <ISO-8601 timestamp>` header comment as the first line (e.g., `# Generated: 2026-03-06T14:30:00Z`). This timestamp is used by /api-status for "Last refresh" display.
>
> Do NOT create browser.ts, MockProvider.tsx, or modify layout.tsx — infrastructure is already in place.
> Do NOT commit. Do NOT use AskUserQuestion."

**Step 4b — Re-run type-generator-agent (conditional, parallel with Step 4a):**

**Condition:** Only run when `web/src/types/api-generated.ts` exists AND the changeset from Step 3 contains `added`, `changed`, or `removed` entries (skip for metadata-only or unchanged results).

Invoke `type-generator-agent` with this prompt:

> "This is a /api-mock-refresh regeneration. Read the updated canonical spec at generated-docs/specs/api-spec.yaml. Regenerate web/src/types/api-generated.ts and web/src/lib/api/endpoints.ts. Run `cd web && npx tsc --noEmit` to verify. Do NOT commit. Do NOT use AskUserQuestion."

Since type-generator writes to `web/src/types/` and `web/src/lib/api/endpoints.ts` while mock-setup writes to `web/src/mocks/`, they have no file conflicts and can run in parallel.

### Step 5 — Report Results

After the agent returns, display a summary to the user:

```
Mock handlers refreshed.

Changes applied:
  Added:   [N] handlers
  Updated: [N] handlers
  Removed: [N] handlers
  Types regenerated: [N] interfaces, [M] endpoint functions

Provenance transitions:
  [N] endpoints transitioned from inferred → user-provided
  [N] user-provided endpoints had schema updates

Reminder: restart your dev server for the service worker to pick up the changes.
  cd web && npm run dev

Tip: run /api-status to see the updated endpoint provenance and coverage.
```

Omit the "Provenance transitions" section if there were no `x-source` changes (e.g., when no source spec update was applied in Step 2). Omit the "Types regenerated" line if Step 4b was skipped (no `api-generated.ts` existed).

---

## Notes

- This command only updates handlers — it does not re-run browser infrastructure setup
- If you need to fully re-initialise the mock layer (e.g. after a `web/src/mocks/` directory was deleted), re-run the DESIGN phase or ask Claude to re-run the mock-setup-agent directly
- The `mock-spec-snapshot.yaml` is the source of truth for "what handlers were last generated from" — do not delete it
